## Context

当前优惠券模型已有 `CouponAudience.PUBLIC` 和 `CouponAudience.MEMBER`，会员专属券通过会员权益领取，但创建订单请求只有 `buyerCouponId` 单值。订单服务只能校验、占用和释放一张买家优惠券，所以会员专属券与普通优惠券不能在同一订单中同时使用。

这次变更需要同时影响优惠券配置、下单金额计算、订单快照、取消/退款释放、前端结算交互和测试。已有“会员折扣”仍按现有规则在优惠券抵扣后的剩余金额上计算，本变更只处理优惠券与优惠券之间的叠加。

## Goals / Non-Goals

**Goals:**

- 允许管理员在优惠券模板上配置该券是否允许与其他券叠加使用。
- 支持买家在一笔订单中同时使用一张普通优惠券和一张会员专属券。
- 当选择两张券时，两张券都必须允许叠加，并且都满足归属、状态、有效期、门槛和 audience 规则。
- 保存订单使用的全部优惠券快照，便于订单详情、后台审计、取消和退款退回。
- 将重要规则映射到自动化测试，覆盖业务逻辑、API 入参和回归路径。

**Non-Goals:**

- 不支持任意数量优惠券叠加；本次只支持一张 `PUBLIC` 与一张 `MEMBER` 的组合。
- 不改变会员折扣规则；会员折扣继续在优惠券抵扣后的金额上计算。
- 不引入跨店铺、品类、商品维度的优惠券适用范围。

## Decisions

### 1. 使用 `Coupon.stackable` 表示是否允许叠加

在 `Coupon` 增加 `stackable` 布尔字段，默认 `false`。管理员创建/编辑优惠券时可设置，响应 DTO 和前端管理页展示该字段。

选择布尔字段而不是枚举，是因为当前需求只有“可叠加/不可叠加”。如果未来需要“仅与会员券叠加”“仅与店铺券叠加”等策略，再迁移为策略枚举。

### 2. 叠加组合限制为一张普通券加一张会员专属券

下单请求新增 `buyerCouponIds`，最多允许 2 个 ID。系统锁定并校验所选买家优惠券后，要求组合中最多一张 `PUBLIC`、最多一张 `MEMBER`。如果只选一张券，则不要求该券 `stackable=true`；如果选两张券，则两张券都必须 `stackable=true`。

保留旧的 `buyerCouponId` 入参用于兼容现有前端或调用方，但如果同时提交 `buyerCouponId` 和 `buyerCouponIds`，应返回 400，避免歧义。

### 3. 优惠券门槛统一按商品原始金额校验

两张券都按订单商品原始金额 `originalAmount` 校验满减门槛，而不是按前一张券抵扣后的金额校验。这样可以避免优惠券选择顺序影响结果。

抵扣金额按固定顺序计算：先普通券，再会员专属券；每张券实际抵扣不超过当前剩余可抵扣金额。订单 `discountAmount` 保存优惠券总抵扣，会员折扣继续基于优惠券抵扣后的金额计算。

### 4. 增加订单优惠券使用明细

新增订单优惠券明细表，例如 `order_coupon_usages`，保存 `order_id`、`coupon_id`、`buyer_coupon_id`、`coupon_name`、`audience`、`stackable`、`threshold_amount`、`coupon_discount_amount`、`applied_discount_amount` 和创建时间。

`Order` 上既有单券字段可在实现阶段选择保留为第一张券的兼容快照，或仅用于旧数据展示；新接口和前端展示应以明细列表为准。

### 5. 占用和释放以优惠券明细为准

创建订单成功后，将所有参与订单的买家优惠券标记为 `USED` 并写入 `usedOrderId`。订单取消、支付超时取消和退款批准成功后，按订单优惠券明细释放该订单占用的全部买家优惠券。

释放仍保持幂等：只释放状态为 `USED` 且 `usedOrderId` 等于当前订单 ID 的买家优惠券，不修改其他订单占用的券。

## Risks / Trade-offs

- [Risk] 老接口仍传 `buyerCouponId`，新接口传 `buyerCouponIds`，可能出现重复或冲突。 → 校验同时提交时报 400，并为单券旧请求保留兼容路径。
- [Risk] 订单折扣明细与订单总金额不一致。 → 金额计算集中在服务层，新增 service 测试校验总抵扣、会员折扣和最终应付金额。
- [Risk] 取消或退款只退回一张券造成资产丢失。 → 退回逻辑改为按订单优惠券明细遍历，并增加多券退回测试。
- [Risk] 历史订单没有优惠券明细。 → 订单详情展示兼容旧单券字段；新订单写入明细。
- [Risk] 叠加配置变更影响已创建订单展示。 → 订单创建时保存 `stackable` 和金额快照，后续模板修改不影响历史订单。

## Migration Plan

- 数据库新增 `coupons.stackable`，历史优惠券默认 `false`。
- 新增订单优惠券明细表，历史订单不回填；展示层兼容旧订单单券字段。
- 前端上线后，管理员可逐步将允许叠加的普通券和会员券配置为 `stackable=true`。
- 回滚时可以保留新增字段和明细表，旧接口 `buyerCouponId` 仍可单券下单。

## Testing Strategy

- 后端 service 测试覆盖：单券兼容、两张可叠加券成功、任一券不可叠加失败、两张同 audience 失败、门槛/状态/归属失败、金额封顶、会员折扣叠加顺序、多券取消和退款释放。
- 后端 controller/API 测试覆盖：创建/编辑优惠券的 `stackable` 字段、下单同时提交新旧字段返回 400、非买家权限保持 403。
- 前端至少运行 `npm.cmd run build`，并手动验证管理员开关、结算页两类券选择和订单详情明细展示。
