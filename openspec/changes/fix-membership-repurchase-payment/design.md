## Context

当前会员购买接口会创建 `membership_purchases` 待支付记录，并返回支付宝支付表单。用户重复点击同一个套餐的购买按钮，或离开支付页后回到会员中心时，购买记录列表会出现 `PENDING_PAYMENT` 记录，但前端没有“继续支付”入口；如果后端复用待支付记录却没有稳定返回支付表单，用户会看到记录存在但无法付款。

这个变更不引入新的支付渠道，也不改变支付成功后会员有效期的续费规则。重点是把待支付会员购买记录作为可继续支付的业务对象处理。

## Goals / Non-Goals

**Goals:**

- 重复购买同一会员套餐时复用已有待支付记录，并稳定返回支付表单。
- 购买记录列表中的待支付会员单可继续支付。
- 前端会员中心支持从“购买”按钮和“购买记录”两条路径打开支付页面。
- 后端避免重复购买、数据唯一冲突等场景落入 500。
- 补充自动化测试和构建验证，覆盖重复购买和继续支付路径。

**Non-Goals:**

- 不新增会员购买取消、超时关闭或自动过期机制。
- 不改变支付宝支付参数结构和支付成功回调逻辑。
- 不改变已支付会员续费的到期时间计算规则。
- 不处理订单支付的继续支付体验。

## Decisions

### 1. 复用待支付会员购买记录，而不是创建新记录

当同一买家对同一套餐已有 `PENDING_PAYMENT` 记录时，购买接口返回这条记录对应的支付表单。这样重复点击“购买”仍能打开支付页面，并避免一组相同待支付记录堆积。

备选方案是每次点击都创建新购买记录。这个方案支付上可行，但会让购买记录列表出现多条待支付记录，并且用户无法判断哪一条应该支付，不利于幂等。

### 2. 增加继续支付接口或等价的后端服务方法

购买记录列表需要针对具体待支付记录发起支付。后端应校验该购买记录属于当前买家，状态必须为 `PENDING_PAYMENT`，并返回支付表单。实现上可以新增 `/api/v1/buyer/membership/purchases/{id}/pay`，也可以复用服务层方法供 controller 调用。

### 3. 前端购买记录中只对待支付记录展示继续支付操作

会员中心购买记录列表对 `PENDING_PAYMENT` 记录展示“继续支付”。已支付、取消或过期记录不展示支付入口。点击后与购买套餐相同，提交隐藏支付表单到新窗口。

### 4. 统一数据冲突响应

如果并发重复请求仍触发数据库唯一约束或数据冲突，后端应返回明确的 400 业务冲突响应，而不是通用 500。业务主路径仍通过复用待支付记录解决。

## Risks / Trade-offs

- [Risk] 复用待支付记录可能复用到过旧的支付单。 → 当前系统尚无会员购买超时机制，本变更不引入过期关闭；后续可单独增加购买单超时。
- [Risk] 用户同时打开多个支付页面可能重复支付。 → 支付成功回调已有幂等要求，同一购买记录最多延长一次会员有效期。
- [Risk] 前端直接提交支付宝表单失败时用户仍停留在会员中心。 → 保持现有购买路径行为，失败时展示错误提示并保留待支付记录供再次点击。

## Testing Strategy

- 后端服务测试覆盖同买家同套餐待支付记录复用。
- 后端 API/controller 测试覆盖继续支付待支付记录成功、访问他人记录返回 403、非待支付记录返回 400。
- 前端运行 `npm.cmd run build` 验证会员中心改动可构建。
- 后端运行 `mvn.cmd test` 验证会员购买、支付回调和相关回归测试。
