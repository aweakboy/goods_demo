## 背景

当前结算页已经收集结构化收货地址，并且后端会通过 `AddressValidationService` 校验地址、保存经纬度和 `VALID` 状态。但地址只存在于订单快照里，买家每次下单都必须重新填写收货人、手机号、省、市、区和详细地址。

本变更新增买家地址簿，作为订单地址快照之前的可复用地址来源。地址簿记录仍然必须通过服务端校验后保存；创建订单时可以引用地址簿记录，也可以继续手动填写地址。

## 目标与非目标

**目标：**

- 买家可以管理自己的常用收货地址，包括新增、编辑、删除和查看。
- 买家可以设置唯一默认地址，结算页优先填入默认地址。
- 结算页可以从地址簿选择地址，自动填入当前下单表单。
- 保存或编辑地址时复用服务端地址校验，成功后保存经纬度和 `VALID` 状态。
- 创建订单时支持 `addressId`，并将地址簿内容复制为订单收货地址快照。
- 地址簿后续修改或删除不得影响历史订单地址快照。
- 自动化测试覆盖地址簿业务规则、权限隔离、默认地址唯一性和使用地址簿下单。

**非目标：**

- 不接入地址搜索联想、行政区级联数据源或地图选点。
- 不做跨买家地址共享。
- 不批量迁移历史订单地址到地址簿。
- 不改变支付、库存扣减、订单状态机或物流流程。
- 不允许前端提交经纬度绕过服务端地址校验。

## 设计决策

### 地址簿使用独立 `BuyerAddress` 表

新增 `BuyerAddress` 或等价实体，按买家保存多条地址。建议字段：

- `id`
- `buyerId`
- `receiverName`
- `receiverPhone`
- `province`
- `city`
- `district`
- `detailAddress`
- `fullAddress`
- `longitude`
- `latitude`
- `addressValidationStatus`
- `formattedAddress`
- `isDefault`
- `createdAt`
- `updatedAt`

独立表比复用订单地址更合适：订单地址是历史快照，不应被后续修改；地址簿是买家的当前偏好，需要独立生命周期。

### 服务端校验仍是权威来源

新增或编辑地址时，后端必须调用 `AddressValidationService`。只有校验成功后，地址簿记录才能保存为 `VALID` 并写入经纬度。

前端只负责展示和提交结构化字段，不得直接写入 `VALID`、经纬度或标准化地址。这样与现有订单地址校验策略保持一致，避免浏览器伪造坐标。

### 默认地址唯一性由服务层保证

同一买家最多只能有一个默认地址。新增第一条地址时可自动设为默认；买家显式设置某条地址为默认时，服务层应在同一事务内取消该买家的其他默认地址。

数据库可以增加 `(buyer_id, is_default)` 的辅助索引，但唯一性主要由服务层事务保障，避免不同数据库对布尔唯一索引语义差异带来实现复杂度。

### 订单创建使用地址快照

`OrderRequest` 新增 `addressId`。创建订单时：

- 如果提供 `addressId`，后端必须确认该地址属于当前买家且状态为 `VALID`，然后复制地址簿字段到订单收货地址快照。
- 如果未提供 `addressId`，继续使用当前手动地址字段，并执行现有地址校验。
- 如果同时提供 `addressId` 和手动地址字段，以 `addressId` 对应地址簿记录为准，避免前端传入不一致数据。

订单创建后只保存快照，不保存对地址簿的强引用作为展示来源。地址簿修改、删除或默认地址变化不影响历史订单。

### 前端结算页以地址选择提升效率

结算页加载买家的地址簿：

- 有默认地址时自动填入默认地址。
- 买家可以从地址列表选择任一地址并填入表单。
- 买家仍可选择手动填写新地址。
- 可提供“保存到地址簿”入口，但保存时必须调用后端地址簿接口并处理校验失败。

如果地址簿为空，结算页保持当前手动填写流程。

## 风险与权衡

- [风险] 地图服务不可用导致新增地址失败 → 与当前下单地址校验保持一致，返回明确提示，避免保存不可定位地址。
- [风险] 默认地址并发设置导致多个默认地址 → 在 service 事务内先清除当前买家的默认地址再设置目标地址，并补充测试。
- [风险] 地址簿被删除后历史订单展示异常 → 订单始终使用地址快照展示，不依赖地址簿记录。
- [风险] 买家访问他人地址 → 所有查询、修改、删除、设置默认和下单引用都必须按 `buyerId` 校验归属。
- [风险] 结算页表单状态复杂 → 地址选择只负责填入已有表单字段；下单提交仍走同一个 `OrderRequest`。

## 迁移计划

1. 新增买家地址表、实体、repository、DTO、service 和 controller。
2. 将地址新增/编辑接入 `AddressValidationService`。
3. 更新 `OrderRequest` 和 `OrderService.createOrder()`，支持 `addressId` 下单。
4. 新增前端地址簿 API 和地址管理/选择组件。
5. 更新结算页加载默认地址、选择地址、手动填写和保存地址流程。
6. 历史订单不迁移；历史订单继续按订单快照或文本地址展示。

## 测试策略

- 后端 service 测试覆盖：新增地址成功、地址校验失败、编辑地址重新校验、删除地址、设置默认地址唯一性、禁止访问他人地址。
- 后端订单测试覆盖：使用地址簿下单保存订单快照、地址簿后续修改不影响订单、引用他人地址返回 403、引用不存在地址返回 404。
- 后端 controller/API 测试覆盖：买家地址接口权限、非买家访问返回 403。
- 前端当前以 `npm.cmd run build` 和手动验证为主；后续有测试基线后应补充结算页地址选择状态测试。
- 必须运行 `backend/` 下的 `mvn.cmd test` 和 `frontend/` 下的 `npm.cmd run build`。

## 待确认问题

- 是否需要独立“我的地址”页面，还是先只在结算页内提供地址管理弹窗。建议实现独立页面，并在结算页复用选择组件。
- 是否需要地址数量上限。建议先限制每个买家最多 20 条地址，避免异常数据膨胀。
