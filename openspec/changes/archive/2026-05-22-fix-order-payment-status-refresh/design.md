## Context

当前支付链路中，`PaymentService.handleNotify()` 依赖支付宝异步通知更新订单状态；`PaymentController.returnPage()` 只把用户重定向回前端结果页。
如果 notify 因内网穿透、签名校验、网络延迟、参数处理或回调异常没有成功落库，用户即使已经在支付宝沙箱扣款，平台订单仍会显示 `PENDING_PAYMENT`。

这不是单纯前端没有刷新，而是支付平台状态与本地订单状态不一致。修复必须在后端建立可信的补偿核验路径，前端只负责触发和展示。

## Goals / Non-Goals

**Goals:**

- 支付平台已经成功扣款时，本地订单最终能稳定更新为 `PAID`。
- notify 成功时继续幂等更新订单状态和支付宝交易号。
- notify 未及时落库时，后端可主动调用支付宝交易查询接口确认交易真实状态，并补偿更新订单。
- 前端支付结果页和订单详情页触发后端核验/刷新，避免用户看到错误的待付款状态。
- 自动化测试覆盖 notify 入账、主动核验入账、金额不匹配、重复处理和失败分支。

**Non-Goals:**

- 不让前端根据 return_url 或页面跳转自行判定支付成功。
- 不用未经支付宝服务端查询确认的浏览器参数直接更新订单为 `PAID`。
- 不引入 WebSocket/SSE 实时推送。
- 不调整退款、发货、超时取消等订单状态机规则。
- 不处理内网穿透域名配置本身，但需要让失败时更容易诊断。

## Decisions

### Decision 1: notify 是主路径，主动交易查询是补偿路径

支付宝异步 notify 仍是首选入账路径；当用户返回平台或页面仍显示待付款时，后端通过 `out_trade_no` 主动查询支付宝交易状态。
如果查询结果为 `TRADE_SUCCESS` 或 `TRADE_FINISHED`，且订单号、金额校验通过，则本地订单补偿更新为 `PAID`。

备选方案是只修前端轮询订单详情，但如果后端从未落库，轮询无法解决问题。
另一个备选方案是在 return_url 中直接标记成功，但 return 请求不能作为支付成功凭证，存在误更新风险。

### Decision 2: 抽出统一的入账方法

`PaymentService` 应抽出统一的 `markOrderPaid` 或等价私有方法，由 notify 和主动查询共同使用。
该方法应具备幂等性：

- `PENDING_PAYMENT` → 更新为 `PAID`，记录 `alipayTradeNo`
- 已 `PAID` → 返回成功，不重复破坏数据
- 其他状态 → 不更新，返回失败或明确结果

这样可以避免 notify 和主动查询两条路径出现状态处理分叉。

### Decision 3: 主动核验接口必须由后端执行支付宝查询

建议新增买家侧接口，例如 `POST /api/v1/orders/{id}/payment/reconcile`。
接口先校验订单属于当前买家，再由后端调用支付宝交易查询；前端只拿到最新订单状态。

查询成功但金额不匹配、订单不存在、订单归属不匹配、订单已取消等情况不得更新为 `PAID`。

### Decision 4: 前端显示围绕“后端真实状态”

`PaymentResultView.vue` 根据 `orderId` 调用支付核验接口；如果仍未确认，再短时间轮询订单详情或重复触发核验。
`OrderDetailView.vue` 发起支付后进入等待状态，并在页面重新获得焦点/可见时触发核验或刷新。

前端文案应区分：

- `PAID`：支付成功
- `PENDING_PAYMENT`：支付确认中
- `CANCELLED` 或其他终态：展示对应状态，不提示成功

### Decision 5: 可测试性优先

当前 `PaymentService` 直接 new `DefaultAlipayClient`，不利于单元测试主动交易查询。
实现时应通过可替换的 client 工厂/包装方法降低测试难度，至少让测试能覆盖交易查询响应、金额校验和状态更新逻辑。

## Risks / Trade-offs

- 支付宝查询接口失败 → 保持订单原状态并提示确认中，允许用户重试，不能擅自改为已付款。
- notify 和主动查询并发更新 → 统一入账方法保持幂等，避免重复写入造成异常。
- 金额精度比较出错 → 使用 `BigDecimal.compareTo`，不要用字符串或浮点比较。
- 订单已被超时取消后才查询到支付成功 → 需要明确处理：本 change 先不改变状态机，遇到非 `PENDING_PAYMENT/PAID` 状态不自动改为 `PAID`，并通过日志暴露人工处理风险。
- 沙箱环境回调不稳定 → 主动查询补偿能降低依赖 notify 成功到达的风险，但仍需要有效支付宝配置。
