## Context

现有订单流程只按购物车商品金额生成订单，平台没有优惠券模型，也没有买家领取和使用记录。优惠券能力会同时影响后台运营、买家资产、结算页金额展示和订单创建事务，因此需要在业务模型、权限边界和订单金额快照上先做明确约束。

优惠券初版定位为平台级满减券：管理员发放，所有买家按规则领取，买家在下单时选择自己已领取且可用的优惠券。暂不支持店铺券、商品券、折扣券、多券叠加和自动推荐最优券。

## Goals / Non-Goals

**Goals:**

- 支持管理员创建、启用、停用和查看优惠券活动。
- 支持买家查看可领取优惠券、领取优惠券、查看自己的优惠券列表。
- 支持买家下单时使用自己的未使用优惠券，按订单商品原始金额校验门槛并抵扣实际支付金额。
- 保存订单级优惠券快照，保证优惠券后续修改、停用或删除不影响历史订单展示、支付和退款金额。
- 用自动化测试覆盖优惠券领取限制、权限、金额计算、重复使用和异常回滚。

**Non-Goals:**

- 不支持商家创建店铺券、指定商品券、折扣券、免邮券或多券叠加。
- 不支持优惠券自动推荐、自动选择最优券或复杂营销活动规则。
- 不处理订单取消、支付超时、退款后的返券策略；初版以下单创建成功为优惠券使用完成点。
- 不引入新的支付渠道或配送费用计算。

## Decisions

### 1. 使用“优惠券模板 + 买家优惠券实例”双表模型

新增 `coupons` 表保存平台发放的优惠券活动，新增 `buyer_coupons` 表保存买家领取后的个人优惠券实例。

`coupons` 关键字段：

- `name`、`description`
- `threshold_amount`：满减门槛
- `discount_amount`：固定抵扣金额
- `total_quantity`、`claimed_quantity`
- `per_user_limit`
- `valid_from`、`valid_to`
- `status`：`DRAFT`、`ACTIVE`、`INACTIVE`
- `created_by`、`created_at`、`updated_at`

`buyer_coupons` 关键字段：

- `buyer_id`、`coupon_id`
- `status`：`UNUSED`、`USED`、`EXPIRED`
- `claimed_at`、`used_at`
- `used_order_id`

原因：优惠券模板负责运营规则和库存，买家优惠券实例负责个人资产和使用状态。这样可以准确表达“同一优惠券活动可被多人领取”，也能在下单时锁定买家的具体券实例，避免仅用 `couponId` 导致归属不清。

备选方案：只建一张 `coupons` 表，在订单中记录买家和使用状态。该方案会把发放规则、库存和个人状态混在一起，难以支持领取列表、个人券包和每人领取限制，因此不采用。

### 2. 下单接口使用 `buyerCouponId`

扩展 `OrderRequest`，新增可选 `buyerCouponId`。买家选择优惠券下单时传入的是自己的买家优惠券实例 ID，而不是优惠券模板 ID。

原因：服务端可以直接校验该实例是否属于当前买家、是否未使用、是否在有效期内、对应模板是否启用，并在同一事务中将实例标记为已使用。

### 3. `Order.totalAmount` 继续表示实际支付金额

订单新增金额和优惠券快照字段：

- `original_amount`：商品原始合计金额
- `discount_amount`：本单总优惠金额
- `coupon_id`、`buyer_coupon_id`
- `coupon_name`
- `coupon_threshold_amount`
- `coupon_discount_amount`

现有 `totalAmount` 保持为实际支付金额，支付、退款和后台统计继续读取该字段。未使用优惠券时，`original_amount = total_amount`，`discount_amount = 0`。

原因：可以最小化支付、退款、订单展示和统计的迁移风险，同时保留足够的审计字段解释订单为什么少付。

### 4. 领取和使用都在服务端事务内校验

领取优惠券时，服务端必须校验：

- 优惠券存在且状态为 `ACTIVE`
- 当前时间在有效期内
- 总领取数量未超过 `total_quantity`
- 当前买家领取数量未超过 `per_user_limit`

使用优惠券下单时，服务端必须校验：

- `buyerCouponId` 存在且属于当前买家
- 买家优惠券状态为 `UNUSED`
- 优惠券模板状态为 `ACTIVE`
- 当前时间在有效期内
- 订单商品原始金额达到 `threshold_amount`

领取库存并发控制优先使用事务内行级锁或条件更新，避免高并发下超发。下单使用优惠券、扣库存、创建订单、删除购物车条目必须位于同一事务，任何一步失败都应回滚优惠券使用状态。

### 5. 前端以“可领取列表、我的优惠券、结算页选择”三处入口实现

管理端新增优惠券页面，用于创建和启停优惠券。买家端新增“优惠券”入口，展示可领取优惠券和我的优惠券。结算页加载买家可用优惠券，并展示商品原价、优惠金额和应付金额。

前端只负责展示和提交选择，不信任前端计算的优惠金额；最终折扣和订单金额以后端返回的订单为准。

## Risks / Trade-offs

- [Risk] 并发领取可能导致优惠券超发 → 在 repository/service 中使用行级锁或原子条件更新，并添加并发或边界测试。
- [Risk] 前端展示金额与后端最终金额不一致 → 前端仅做预览，提交后以订单详情返回金额为准；后端测试覆盖最终金额。
- [Risk] 优惠券下单失败后状态被错误标记为已使用 → 订单创建和优惠券使用状态更新放在同一事务，失败路径测试验证不创建订单、不扣库存、不占用券。
- [Risk] 历史订单受优惠券后续修改影响 → 订单保存优惠券名称、门槛、抵扣金额等快照字段，详情页读取订单快照。
- [Risk] 初版不返券可能影响用户体验 → 在产品文案和后续变更中明确返券策略，本变更只保证下单创建成功后券不可重复使用。

## Testing Strategy

- 后端 service 测试覆盖优惠券创建、启停、领取成功、库存不足、超出每人领取限制、过期/停用不可领取。
- 后端 service 测试覆盖下单使用优惠券成功、未达门槛、他人优惠券、已使用优惠券、过期优惠券、异常回滚。
- 后端 controller/API 测试覆盖管理员接口权限、买家领取接口权限、非买家/非管理员返回 403。
- 前端当前没有测试 runner，前端变更至少运行 `npm.cmd run build`，并保留明确手动验证任务。
- 整体后端验证运行 `mvn.cmd test`，OpenSpec 变更验证运行 `openspec validate add-coupon-lifecycle`。
