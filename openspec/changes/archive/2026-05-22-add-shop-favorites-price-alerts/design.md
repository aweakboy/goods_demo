## Context

当前系统已经具备店铺主页、店铺搜索、商品详情、卖家商品编辑和买家登录能力，但买家没有保存感兴趣店铺的入口，也无法对关注商品设置目标价。现有通知能力尚未形成通用站内通知模型，因此降价提醒需要同时补齐通知数据模型和买家通知中心。

本变更跨越后端业务模型、商品价格更新链路、买家端页面和 API wrapper。实现时应保持与现有 Spring Boot + JPA + Vue + Element Plus 风格一致，并复用现有角色权限约束：收藏、提醒和通知仅面向 `BUYER`。

## Goals / Non-Goals

**Goals:**

- 买家可以收藏和取消收藏 ACTIVE 店铺，并查看自己的收藏店铺列表。
- 店铺主页返回当前买家是否已收藏该店铺，同时展示收藏数量。
- 买家可以对 ACTIVE 商品设置目标价，目标价到达后收到站内降价通知。
- 商品价格从旧价下降到小于或等于目标价时，系统触发一次通知并关闭该提醒，避免重复通知。
- 买家可以查看降价通知、未读数量，并将单条或全部通知标记为已读。
- OpenSpec artifacts 明确包含自动化测试和验证命令。

**Non-Goals:**

- 不实现短信、邮件、微信模板消息、浏览器 Push 等外部通知渠道。
- 不实现复杂推荐、营销活动、价格趋势图或历史价格分析。
- 不实现跨设备实时 WebSocket 推送；通知通过买家主动拉取展示。
- 不改变卖家创建/编辑商品的既有权限边界。

## Decisions

### 1. 使用独立收藏表，而不是在 Shop 或 User 上保存集合

新增 `shop_favorites` 表，字段包含 `buyer_id`、`shop_id`、`created_at`，并对 `(buyer_id, shop_id)` 建唯一约束。

理由：收藏是买家和店铺之间的多对多关系，独立表便于分页查询、计数、权限隔离和幂等处理。将收藏 ID 列表放在用户或店铺记录中会增加并发更新和分页成本。

备选方案：在 `shops` 上维护收藏人列表或计数字段。计数字段可作为后续性能优化，但本次不作为唯一事实来源，避免数据一致性问题。

### 2. 商品降价提醒采用“一买家一商品一条提醒记录”

新增 `price_alerts` 表，字段包含 `buyer_id`、`product_id`、`target_price`、`status`、`last_notified_price`、`triggered_at`、`created_at`、`updated_at`，并对 `(buyer_id, product_id)` 建唯一约束。

当买家再次对同一商品设置提醒时，系统更新原记录并重置为 `ACTIVE`，而不是创建多条提醒。

理由：买家自然期望同一商品只有一个当前目标价；唯一约束可以防止重复提醒和并发重复创建。

备选方案：保留提醒历史多条记录。该方案便于分析价格兴趣，但会增加触发去重复杂度，本期价值不足。

### 3. 降价检测在商品价格保存后同步执行

在卖家或管理员成功更新商品价格后，如果新价格低于旧价格，则调用 `PriceAlertService.processPriceChange(productId, oldPrice, newPrice)`。服务查询该商品 `ACTIVE` 且 `target_price >= newPrice` 的提醒，生成通知并将提醒状态改为 `TRIGGERED`。

理由：当前系统没有消息队列或任务调度基线，同步处理实现成本低、行为确定，测试容易覆盖。

备选方案：使用异步事件或定时任务扫描。该方案更适合大量提醒，但需要引入事件投递、幂等重试和监控，本期不引入。

### 4. 站内通知使用通用买家通知表

新增 `buyer_notifications` 表，字段包含 `buyer_id`、`type`、`title`、`content`、`product_id`、`shop_id`、`read_at`、`created_at`。降价通知类型为 `PRICE_DROP`。

理由：降价提醒只是第一类通知，独立通知表可复用给后续订单、物流、优惠券等站内消息，且便于统一未读数和已读状态。

备选方案：只在 `price_alerts` 表保存触发状态，不建通知表。该方案无法提供稳定通知中心，也不利于后续扩展。

### 5. 前端采用主动拉取，不引入实时推送

买家导航展示通知入口和未读数量；进入通知中心时分页拉取通知。商品详情页展示设置/更新目标价入口；店铺主页展示收藏按钮和收藏状态。

理由：主动拉取能覆盖当前业务目标，避免 WebSocket 或 SSE 的连接管理成本。

## Risks / Trade-offs

- [Risk] 热门商品可能存在大量降价提醒，同步触发会拉长商品编辑请求耗时。→ 通过 repository 查询只筛选命中提醒；后续如数据量增长，再迁移为异步任务。
- [Risk] 商品价格多次更新可能重复通知。→ 使用提醒状态 `TRIGGERED` 和唯一提醒记录，单次提醒只触发一次；买家重新设置目标价才会再次激活。
- [Risk] 店铺收藏数量实时统计可能增加查询成本。→ 初期使用聚合查询；如果后续性能不足，再引入冗余计数字段并用事务维护。
- [Risk] 用户可能收藏后店铺被禁用。→ 收藏记录保留；收藏列表默认不展示已禁用店铺或标记为已关闭，避免进入不可访问页面。
- [Risk] 前端没有自动化测试基线。→ 后端业务与 API 使用自动化测试覆盖；前端通过 `npm.cmd run build` 和明确手动验证覆盖交互。

## Migration Plan

1. 在 `schema.sql` 中新增 `shop_favorites`、`price_alerts`、`buyer_notifications` 表和必要索引/唯一约束。
2. 后端新增实体、枚举、repository、service、controller 和 DTO。
3. 在商品价格更新成功后接入降价提醒处理。
4. 前端新增收藏、目标价提醒、通知中心 API wrapper 和页面入口。
5. 运行 `mvn.cmd test`、`npm.cmd run build` 和 `openspec validate add-shop-favorites-price-alerts`。

回滚时可移除前端入口和新增 API 调用；数据库新增表不影响既有订单、商品和店铺主流程，可保留为空表或后续清理。

## Testing Strategy

- 后端 service 测试覆盖收藏幂等、取消收藏幂等、收藏列表权限隔离、目标价校验、重复设置更新原提醒、降价触发通知和重复触发防护。
- 后端 controller 或 WebMvc 测试覆盖 `BUYER` 权限、非买家访问返回 403、收藏/提醒/通知接口响应结构。
- 商品价格更新相关测试覆盖卖家更新商品价格后调用降价提醒处理，以及价格未下降时不触发通知。
- 前端当前无测试 runner，变更需运行 `npm.cmd run build`，并手动验证店铺主页收藏按钮、商品详情目标价弹窗、通知中心未读/已读状态。
