## 背景与动机

当前物流地图只用发货地和收货地做直线模拟，不能展示真实道路路线、行驶距离或预计耗时。为了在不把路径规划调用量绑定到页面访问量的前提下提供更真实的物流体验，需要引入稳定调用模型：在发货或地址路线关键数据变化时规划一次并缓存路线快照，订单页面只读取已保存结果。

## 变更内容

- 新增高德路径规划能力：后端在 shipment 创建或明确刷新路线时调用路径规划服务。
- 新增路线快照持久化：保存路线供应商、起终点、路线 polyline、距离、预计耗时、规划时间、规划状态和失败原因。
- 更新物流地图数据：优先返回真实路线快照和基于路线距离的当前位置；无快照或规划失败时降级到现有直线模拟。
- 更新买家、卖家、管理员订单页面：在路线快照可用时展示道路路线、已完成路线段、当前位置、距离和预计耗时。
- 新增调用保护：避免每次打开订单详情触发路径规划；支持配置开关、超时、失败降级和必要的管理员手动刷新入口。
- 保留兼容：未配置高德 Web 服务 Key、配额耗尽、路径规划失败、缺少坐标或历史物流单号时继续展示文本轨迹或现有模拟路线。

## 能力范围

### 新增能力

- `shipment-route-planning`: shipment 路径规划调用、路线快照缓存、路线状态、失败降级和基于路线的当前位置计算。

### 修改能力

- `order-management`: 买家订单详情在物流具备路线快照时展示真实道路路线、距离、预计耗时和当前位置。
- `seller-dashboard`: 卖家订单管理在发货后展示缓存路线快照，并在路线不可用时保持物流操作可用。
- `admin-order-oversight`: 管理员订单详情展示路线快照，并支持在必要时手动刷新 shipment 路线规划。

## 影响范围

- 后端：新增路径规划配置、客户端接口、高德路径规划实现、路线快照实体/字段、路线规划 service、DTO 扩展和测试。
- 前端：扩展 `ShipmentMap.vue` 以渲染路线 polyline、已完成路线段、距离/预计耗时；更新买家、卖家、管理员订单页面展示路线状态和刷新结果。
- 数据库：需要新增 shipment route snapshot 表或在 shipment 上增加路线快照字段；优先使用独立表以隔离较大的 polyline 数据。
- 配置：继续区分后端高德 Web 服务 Key 与前端 JS API Key；新增路径规划启用开关、超时和刷新保护配置。
- 外部系统：高德 Web 服务路径规划 API；调用量由发货/刷新次数驱动，不随订单详情页面访问次数线性增长。
- 依赖关系：建议在 `integrate-address-validation-map-service`、`upgrade-shipping-model` 和 `map-logistics-simulation` 的核心实现稳定后应用。
