## 背景

当前系统已经有两层地图能力：`integrate-address-validation-map-service` 负责通过高德 Web 服务校验地址并保存经纬度；`map-logistics-simulation` 负责用 shipment 的发货地/收货地坐标生成直线模拟路线和当前位置。用户现在需要真实路径规划，但路径规划 API 的调用量不应该随买家、卖家或管理员打开订单页面的次数增长。

本设计把路径规划定位为 shipment 的增值快照数据：发货时或管理员明确刷新时调用一次高德路径规划，保存路线结果；后续订单页面只读取缓存快照并绘制地图。路径规划失败时不阻断发货和订单查看，继续使用现有直线模拟与物流时间线。

## 目标与非目标

**目标：**

- 在 shipment 创建时，基于发货地和收货地经纬度调用路径规划并保存路线快照。
- 页面展示订单详情时只读取已保存路线快照，不触发外部路径规划调用。
- 路线快照可用时展示真实道路 polyline、总距离、预计耗时、已完成路段和当前位置。
- 路线规划不可用、失败、配额耗尽、缺少坐标或未配置 Key 时，发货流程和订单查看继续可用。
- 管理员可以在必要时手动刷新路线规划，并受冷却时间或防重复调用保护。
- 自动化测试覆盖路径规划成功、失败、跳过、缓存读取、刷新保护和页面降级相关逻辑。

**非目标：**

- 不接入真实快递公司轨迹或车辆实时位置。
- 不做多包裹、多发货点、多途经点或复杂配送调度。
- 不在每次订单详情查询时重新规划路线。
- 不把高德 Web 服务 Key 暴露给前端。
- 不在本期实现异步队列或后台重试任务；如果同步调用的延迟不可接受，后续可以把规划任务异步化。

## 设计决策

### 路径规划调用绑定 shipment 生命周期，不绑定页面访问

后端只在两个入口调用路径规划：

- 卖家创建 shipment 后，坐标完整且路径规划启用时自动规划一次。
- 管理员在订单详情中点击刷新路线时显式重新规划。

买家订单详情、卖家订单列表、管理员订单详情等读取接口不得调用路径规划服务。它们只读取 `ShipmentRouteSnapshot` 或等价快照数据，并组合进 shipment 地图响应。

备选方案是在页面加载时发现无路线就即时规划。该方案实现简单，但页面访问量会直接变成外部 API 调用量，成本和失败面不可控，因此不采用。

### 路线快照使用独立表持久化

新增 `ShipmentRouteSnapshot` 或等价实体，按 shipment 一对一保存路线规划结果。建议字段：

- `shipmentId`
- `provider`，如 `AMAP`
- `planningStatus`，如 `AVAILABLE`、`FAILED`、`SKIPPED`
- `originLongitude`、`originLatitude`、`destinationLongitude`、`destinationLatitude`
- `routePolylineJson`，保存标准化后的坐标数组，而不是直接把 provider 原始字符串暴露给前端
- `distanceMeters`
- `durationSeconds`
- `plannedAt`
- `failureReason`
- `lastRefreshRequestedAt`

独立表比把 polyline 塞进 `shipments` 更清晰：路线数据体积较大、生命周期和失败状态独立，也便于后续替换 provider 或增加刷新历史。

### 供应商适配器隔离高德路径规划细节

新增 `RoutePlanningClient` 接口和 `AmapRoutePlanningClient` 实现。业务服务只依赖标准化结果：

- 路线点数组
- 距离米数
- 预计耗时秒数
- provider 状态和错误信息

高德响应解析、超时、认证失败、配额限制和空路线都在适配器层转换为明确异常或失败结果。这样后续切换供应商或增加 mock 供应商不影响 shipment 业务逻辑。

### 发货流程失败开放，路线规划失败只影响地图增强

创建 shipment 时如果路径规划失败，系统 SHALL 继续完成发货、记录 shipment 事件并返回订单。路线快照保存为 `FAILED` 或 `SKIPPED`，响应里 `routeSource` 回退为 `SIMULATED`，并带上规划失败原因供页面展示。

备选方案是路径规划失败则发货失败。该方案能保证每个已发货订单都有真实路线，但会让外部地图服务成为核心交易流程的单点依赖，不适合当前项目。

### 地图响应保留现有模拟接口并增加路线来源

继续复用 `ShipmentResponse.mapSimulation` 或等价字段，向后兼容已有页面。扩展字段建议包括：

- `routeSource`: `PLANNED` 或 `SIMULATED`
- `routePath`: 完整路线点数组，真实路线可用时为道路路线，否则为起终点直线
- `completedPath`: 从起点到当前位置的已完成路线点数组
- `distanceMeters`
- `durationSeconds`
- `plannedAt`
- `planningStatus`
- `planningFailureReason`

后端负责基于 shipment 状态和路线点计算当前位置及已完成路段。前端只渲染完整路线、完成路线和 marker，避免每个页面重复几何计算。

### 基于路线距离计算当前位置

路线快照可用时，当前位置不再用经纬度直线插值，而是按 shipment 状态映射进度，再沿路线 polyline 的累计距离取点：

- `SHIPPED`: 0.05
- `IN_TRANSIT`: 0.50
- `OUT_FOR_DELIVERY`: 0.85
- `DELIVERED`: 1.00
- `EXCEPTION`: 使用最近可计算进度，最低要求回退到 0.50

这保留现有状态驱动模型，同时让当前位置落在真实道路路线附近。真实物流位置仍然不是本期目标。

### 管理员刷新受保护

管理员刷新路线只更新路线快照，不改变 shipment 状态或轨迹事件。刷新应受最小间隔配置保护，例如 `shipping.route-planning.min-refresh-interval-minutes`。如果已有可用快照且刷新失败，系统保留旧快照并返回失败原因，避免一次失败把可展示路线降级掉。

## 风险与权衡

- [风险] 高德路径规划配额耗尽或 Key 配置错误 → 保存失败状态并回退到模拟路线；日志记录 provider 错误；页面显示降级原因。
- [风险] 发货时同步规划增加响应耗时 → 设置短超时；失败开放；后续如需要再改为异步任务。
- [风险] Polyline 数据较大导致订单列表响应变重 → 卖家列表可返回摘要或按现有页面需求返回；如列表性能受影响，后续拆出详情接口或懒加载路线。
- [风险] 不同 provider 坐标系或 polyline 格式不一致 → adapter 层输出标准化经纬度数组；前端不解析 provider 原始格式。
- [风险] 管理员误点刷新造成额外调用 → 冷却时间和权限校验保护；读取接口绝不触发规划。
- [风险] 旧数据没有路线快照 → 保持现有直线模拟和文本时间线，不需要数据迁移。

## 迁移计划

1. 添加路线规划配置和路线规划供应商适配器。
2. 新增路线快照实体、repository、状态枚举和 DTO。
3. 将 shipment 创建流程接入一次性路线规划，失败时保存失败/跳过状态。
4. 扩展 shipment 地图响应，优先使用路线快照，缺失时回退现有模拟。
5. 增加管理员刷新路线接口和前端按钮。
6. 扩展 `ShipmentMap.vue` 渲染 routePath、completedPath、距离和预计耗时。
7. 对历史 shipment 不做强制补算；管理员可按需刷新单笔订单路线。

回滚策略：保留数据库快照表不影响核心订单；关闭 `shipping.route-planning.enabled` 后系统停止新增外部路径规划调用，并继续返回模拟路线。

## 测试策略

- 后端 service 测试覆盖：坐标完整时创建可用路线快照、缺坐标跳过且不调用供应商、供应商异常不阻断发货、读取订单详情不触发供应商、路线进度沿 polyline 计算当前位置、刷新冷却和刷新失败保留旧快照。
- 后端 client 测试覆盖：高德成功响应解析、空路线、认证失败/配额失败、HTTP 异常和超时转换。
- 后端 controller/API 测试覆盖：管理员刷新路线权限、成功返回最新 shipment 地图数据、非管理员刷新返回 403。
- 前端验证覆盖：有路线快照时展示道路路线和距离/预计耗时；规划失败或未配置 Key 时降级；管理员刷新后地图更新。
- 必须运行 `backend/` 下的 `mvn.cmd test` 和 `frontend/` 下的 `npm.cmd run build`。

## 待确认问题

- 路线规划默认使用驾车路线，是否需要配置高德策略参数，例如速度优先、费用优先或躲避拥堵。
- 卖家订单列表是否需要完整 routePath，还是只在展开/详情时加载完整路线以减小响应体。
