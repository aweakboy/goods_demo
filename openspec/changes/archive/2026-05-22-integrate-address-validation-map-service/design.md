## Context

`structured-shop-order-addresses` 已经为店铺和订单收货地址建立结构化字段，并预留 `longitude`、`latitude`、`addressValidationStatus`。但这些字段目前仍主要来自用户输入，默认 `UNVERIFIED`，系统不能判断地址是否真实存在，也不能可靠展示地图定位。

本 change 在此基础上接入地图服务。考虑到项目面向中文地址和本地开发，默认选择高德地图：

- 后端使用高德 Web 服务进行地理编码和地址校验，避免把 Web 服务 Key 暴露给浏览器。
- 前端使用高德 JS API 只负责地图展示和交互预览。
- 业务保存的是服务端校验后的经纬度和校验状态，前端不得绕过服务端直接写入 `VALID`。

## Goals / Non-Goals

**Goals:**

- 店铺地址和订单收货地址提交时通过服务端地理编码校验。
- 成功校验后保存经纬度，并将地址校验状态更新为 `VALID`。
- 地址无匹配、行政区不一致或坐标缺失时拒绝保存/下单，并提示用户修正。
- 店铺主页、订单详情、卖家订单和管理员订单详情在有经纬度时展示地图定位。
- 为后续距离估算、发货模拟和路线规划保留稳定的经纬度数据。
- 后端测试覆盖地址校验成功、失败、外部服务异常和业务保存路径。

**Non-Goals:**

- 不做完整物流轨迹、路线规划、运费计算或配送时效估算。
- 不做用户常用地址簿。
- 不抽象多地图供应商 UI；本期只实现一个 provider adapter，默认高德。
- 不把高德 Web 服务 Key 暴露到前端。
- 不强制迁移历史 `UNVERIFIED` 地址，历史数据仍按文本展示。

## Decisions

### Decision 1: 服务端校验是权威来源

前端可以展示地图和发起预校验，但最终保存店铺地址或创建订单时，后端必须重新调用地图服务校验地址。只有后端校验成功的结果才能写入经纬度和 `VALID` 状态。

备选方案是前端直接调用地图 JS API 后把坐标提交给后端，但浏览器数据可被篡改，也难以保证服务端业务规则一致。

### Decision 2: 使用结构化地址组装查询文本

校验时使用 `province + city + district + detailAddress` 作为地理编码文本，并把 `city` 或行政区作为辅助限制条件。返回结果必须满足：

- 有经纬度。
- 返回的省/市/区与用户提交字段一致或可明确归一。
- 地址匹配精度不能低到只匹配城市级；至少需要区县或道路/门址级别。

如果返回多条结果，优先选择行政区完全匹配且精度最高的一条；若仍无法唯一判断，返回“地址无法确认”。

### Decision 3: 外部服务失败时失败关闭

对新注册/编辑店铺和新订单，地图服务不可用时不保存为 `VALID`，并返回“地址校验暂不可用，请稍后重试”。这会牺牲少量可用性，但避免产生无法定位的新增业务数据。

历史地址不受影响，仍可按文本展示。

### Decision 4: 复用现有地址字段，少量扩展响应

实体上优先复用已有字段：

- 店铺：`longitude`、`latitude`、`addressValidationStatus`
- 订单收货地址快照：`receiverLongitude`、`receiverLatitude`、`receiverAddressValidationStatus`

如实现中需要展示地图供应商返回的标准化地址，可在响应 DTO 中添加非持久化或持久化的 `formattedAddress`/`adcode` 字段；但本 change 的最小落地不要求新增数据库字段。

### Decision 5: 前端地图封装成可复用组件

新增一个轻量地图组件或 composable，用于加载高德 JS API、渲染地图、展示 marker。业务页面只传入经纬度、标题和地址文本，避免每个页面重复写地图加载逻辑。

地图组件必须处理以下状态：

- 未配置 JS API Key：展示文本地址和配置提示。
- 无经纬度：不渲染地图，只展示文本地址。
- 地图加载失败：展示文本地址和失败提示。

## Risks / Trade-offs

- 外部服务网络失败或配额耗尽 → 后端返回明确错误，新地址提交失败；日志记录 provider 错误，前端提示稍后重试。
- 中文行政区名称存在别名或辖区调整 → 采用宽松归一规则，但仍要求能匹配到合理区县；无法确认时让用户重新选择。
- 高德 Key 类型配置错误 → 将 Web 服务 Key 和 JS API Key 分开配置，并在启动/调用时给出清晰错误。
- 前端地图 Key 暴露 → JS API Key 不用于服务端校验；敏感 Web 服务 Key 只放后端。
- 历史地址没有经纬度 → 历史记录继续展示文本，必要时后续单独做批量补全 change。
- 多卖家订单未来需要多发货点 → 本期只定位店铺地址和订单收货地址，不拆包、不规划路线。

## Migration Plan

1. 添加地图配置和后端地址校验服务。
2. 将店铺注册/编辑、订单创建接入服务端校验。
3. 更新前端表单为“校验后提交”的交互，展示地图预览。
4. 在店铺主页和订单相关详情页渲染地图 marker。
5. 保留历史 `UNVERIFIED` 数据，不做强制迁移。
